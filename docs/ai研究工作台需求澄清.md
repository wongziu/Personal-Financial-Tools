# AI 研究工作台需求澄清

更新时间：2026-06-08

## 澄清方式

本文件模拟多轮需求澄清。由于本轮目标要求“多轮自我需求澄清”，且没有要求停下来等待用户确认，因此这里把不确定点、判断依据、最终决策写成可追溯记录。

## 第 1 轮：产品边界

问题：AI 研究工作台要不要直接生成交易？

判断：

- 当前系统已有 `trade_decisions` 和风控/例外模块。
- 开源项目里 TradingAgents 和 AI Hedge Fund 都有交易/组合经理概念，但也都强调教育、研究和风险提示。
- 本地系统定位是投资决策记录和复核，不是自动交易系统。

决策：

- AI 只输出可审查分析、风险边界和交易决策草案建议。
- 不自动写入交易流水，不自动执行订单。

## 第 2 轮：agent 工作流复杂度

问题：是否需要引入 LangGraph、AutoGen 或外部 agent 框架？

判断：

- 当前项目是 Next.js + SQLite 本地单用户系统，没有 Python agent runtime。
- 引入外部 agent 框架会增加运行时、部署、测试和模型配置复杂度。
- 本轮更重要的是把协作结构和可审查产物落地。

决策：

- 先实现 TypeScript 轻量顺序 agent workflow。
- 数据结构保留 `stageId`、`role`、`status`、`inputSummary`、`output`、`latencyMs`，未来可以迁移到图执行器。

## 第 3 轮：agent 角色

问题：采用哪些 agent？

候选：

- TradingAgents 风格：基本面、技术、新闻、舆情、牛熊研究员、交易员、风控、组合经理。
- AI Hedge Fund 风格：投资大师人格 + 分析员 + 风控 + 组合经理。
- 本地工作台风格：围绕已有数据表拆解。

决策：

本轮采用 5 个 stage：

1. `evidence`：证据分析员，检查信息来源质量、缺口和冲突。
2. `thesis`：论点审查员，判断本地论点是否被支持、削弱或需要复核。
3. `risk`：风险官，识别风险、触发条件和仓位/流动性限制。
4. `decision`：决策主席，提出可审查行动建议，不直接执行。
5. `critic`：复核员，检查前四步是否有编造、证据不足、遗漏问题。

## 第 4 轮：模型调用策略

问题：每个 agent 单独调用模型，还是一次 prompt 模拟多个 agent？

判断：

- 单次 prompt 成本低，但中间阶段不可独立失败、不可单独重跑。
- 多次调用更符合 agent workflow，也能保留阶段耗时和错误状态。
- 当前模型网关已经跑通 OpenAI-compatible Chat Completions。

决策：

- 初期每个 stage 单独调用模型。
- 如果某个 stage 失败，返回已完成 stage 和失败状态，不伪造结果。
- UI 显示每个 stage 的产出，避免只给最终结论。

## 第 5 轮：数据来源

问题：是否联网抓取实时新闻和行情？

判断：

- 目标调研提到开源项目使用多数据源和实时新闻。
- 当前系统的可信数据边界是本地 SQLite 记录、手工录入价格、信息来源和研究事件。
- 联网抓取会引入 API key、合规、数据质量和测试复杂度。

决策：

- 本轮 agent workflow 只使用本地研究记录。
- 输出必须提示缺少哪些外部信息或下一步应该补哪些来源。
- 后续迭代再规划数据源适配层。

## 第 6 轮：结果落地

问题：agent workflow 的结果是否需要入库？

判断：

- 当前没有 AI 分析结果表。
- 直接建表会扩大迁移和历史查看范围。
- 目标要求实现整体功能，但没有强制持久化每次 AI 运行。

决策：

- 本轮结果只在 UI 展示，不入库。
- 结构设计保持可序列化，后续可新增 `research_agent_runs` 和 `research_agent_stages`。

## 第 7 轮：验收标准

验收标准：

- 本地存在调研规划、需求澄清、完整改造计划三份 Markdown。
- 后端有可测试的 agent workflow 编排函数。
- API 能运行 agent workflow 并返回 stage 列表。
- UI 能选择标的和问题，触发 agent workflow，展示阶段结果。
- 单测、类型检查、lint、build、E2E 全部通过。
- 分支合并到 `main` 并推送 GitHub。
